如果你**就是要做 LogFormer**，那其实也完全可以，而且从课程设计角度来说还有一个优势：

> 你可以把老师要求的 Prometheus + Grafana 当作监控系统使用，把 LogFormer 当作异常检测算法复现。

这两件事并不冲突。

---

# 你需要修改目标

不要说：

> 用 Prometheus 指标训练 LogFormer

因为 LogFormer不是干这个的。

而应该说：

> 利用 Chaos Mesh 构造故障场景，通过 Prometheus+Grafana 监控系统观察系统状态，同时采集微服务日志，使用 LogFormer 进行日志异常检测。

这样逻辑就通顺了。

---

# 你的实验架构应该变成

```text
Online Boutique
        │
        ▼
   Chaos Mesh
        │
        ▼
故障注入
(PodKill/Delay/Loss)
        │
 ┌──────┴──────┐
 ▼             ▼
Prometheus    Logs
Grafana        │
               ▼
         LogFormer
               ▼
        Anomaly Score
```

其中：

Prometheus负责：

* CPU
* Memory
* Latency

用于验证故障是否真正发生。

LogFormer负责：

* 检测日志异常。

# 你最需要解决的问题

## 问题1：如何采集大量日志

单纯：

```bash
kubectl logs
```

不够。

建议安装 Loki。

架构：

```text
Pod
 ↓
Promtail
 ↓
Loki
 ↓
Grafana
```

原因：

Online Boutique有十几个服务：

```text
frontend
checkoutservice
cartservice
paymentservice
emailservice
shippingservice
...
```

故障发生时：

```text
frontend timeout
↓
checkout failed
↓
payment retry
```

日志会分散在多个Pod。

Loki会帮你统一收集。

# 问题3：如何得到LogFormer输入

LogFormer不是直接吃原始日志。

论文流程：

```text
Raw Log
 ↓
Drain
 ↓
Template
 ↓
Sequence
 ↓
LogFormer
```

例如：

原始日志：

```text
User 123 login success
User 456 login success
User 789 login success
```

Drain：

```text
User <*> login success
```

模板ID：

```text
E1
```

最终：

```text
E1 E1 E1 E1 E2 E3 E1 ...
```

送入Transformer。
